BAJO Fire-and-forget sin manejo de excepciones PedidoDocumentoPage.xaml.cs:94–96 ▼ 
_ = HidratarSelectorClienteAsync(relId);
_ = ViewModel.CargarConfiguracionFiscalAsync();

Si estas operaciones fallan (red, timeout, BD inaccesible), el error se pierde silenciosamente. El usuario no recibe feedback y la UI queda parcialmente hidratada.

Envolver en método helper con try-catch que propague el error al ViewModel.ErrorMessage, o usar SafeFireAndForgetAsync() con callback de error.
✅

✅

EF Core & Persistencia

Queries N+1, Include faltantes, HasDefaultValue, configuraciones
ALTO N+1 Query — PedidoService.ListarAsync PedidoService.cs:45–54 ▼ 
var lista = await query
    .OrderByDescending(p => p.Fecha).ThenByDescending(p => p.Id)
    .Select(p => new {
        Pedido          = p,
        FolioCotizacion = p.CotizacionId != null
            ? _context.Cotizaciones                     // ← Subquery por cada pedido
                .Where(c => c.Id == p.CotizacionId)
                .Select(c => c.Folio).FirstOrDefault()
            : null
    })
    .ToListAsync(ct);

Por cada Pedido con CotizacionId != null, EF Core genera una query SQL adicional. Con 100 pedidos vinculados a cotizaciones: 101 queries en lugar de 1-2. Impacto medible en respuesta del módulo Pedidos con volumen real.

// Solución: LEFT JOIN explícito
var lista = await query
    .Include(p => p.Cotizacion)        // ← Una sola query con JOIN
    .OrderByDescending(p => p.Fecha)
    .ThenByDescending(p => p.Id)
    .ToListAsync(ct);

// En MapToResumen:
MapToResumen(p, p.Cotizacion?.Folio)
✅
HasDefaultValue correctamente alineados. Todos los HasDefaultValue en EF configurations usan el tipo CLR correcto de la propiedad (no el tipo SQL convertido). Después del fix EstadoFinancieroPedido.SinPago, ningún default tipado incorrectamente.
✅
Foreign Keys actualizadas. ventas.Venta, ventas.OrdenTrabajo, ventas.Factura y ventas.Pedido ahora apuntan correctamente a core.RelacionComercial. FKs duplicadas en Factura eliminadas.
MEDIO SucursalId ?? 1 hardcodeado como fallback VentaDocumentalService.cs:168 ▼ 
SucursalId: pedido.SucursalId ?? 1,  // ← Asume SucursalId=1 si el Pedido no tiene sucursal

Si la empresa tiene múltiples sucursales y el Pedido no tiene SucursalId asignado, la Venta se crea en Sucursal 1 incorrectamente. En entornos multi-sucursal, puede causar reporting inconsistente.

Validar explícitamente: si pedido.SucursalId es null, usar la sucursal activa del SessionService o retornar error descriptivo.

5. Agregar appsettings.*.json al .gitignore.

MEDIO Sin permiso formal para operaciones de Anticipo (ADR-065) PermisosClave.cs · PedidoService.cs:350 ▼

La feature de Anticipos (ADR-065) reutiliza pedido.editar para registrar anticipos. Esto es funcionalmente correcto pero semánticamente impreciso: un usuario con permiso de editar pedidos puede registrar cobros, que es una responsabilidad financiera distinta.

// PermisosClave.cs — sin categoría Anticipo
// PedidoService.RegistrarAnticipoAsync:
if (!await _auth.PuedeAsync(PermisosClave.Pedido.Editar, ct))  // ← Reutilizado
Agregar en PermisosClave.cs: public static class Anticipo { public const string Registrar = "anticipo.registrar"; } y registrar el permiso en el seed de BD (51 permisos → 52). Permite control de acceso granular: un cajero puede registrar anticipos sin poder editar líneas del pedido.
